## Une architecture de confiance

DIGIYLYFE doit être une architecture métiers pro, mais aussi une architecture de protection.

Le numérique digne ne cherche pas à tout savoir sur les gens.  
Il donne aux professionnels les bons outils, avec les bonnes limites.

---

## Phrase centrale

**DIGIYLYFE ne compile pas les personnes.  
DIGIYLYFE équipe les professionnels avec leur accord.**

---

## Doctrine

DIGIY ne doit jamais devenir un aspirateur à données.

La donnée doit rester :

- utile ;
- limitée ;
- consentie ;
- protégée ;
- séparée selon les espaces ;
- visible seulement quand elle doit l’être ;
- validée quand elle devient sensible.

---

## Séparation des espaces

### Public

Ce qui peut être public :

- nom professionnel ;
- métier ;
- zone ;
- fiche ;
- photos choisies ;
- description ;
- lien de contact choisi ;
- QR public ;
- services ou produits publiés volontairement.

### Pro

Ce qui reste privé au pro :

- cockpit ;
- notes ;
- clients ;
- historiques ;
- actions métier ;
- données de gestion ;
- informations PAY ;
- planning interne ;
- preuves de paiement.

### Admin

Ce qui reste admin :

- validations ;
- activations ;
- traces ;
- preuves ;
- abonnements ;
- liens privés ;
- informations de contrôle ;
- alertes sécurité.

### Fondateur

Ce qui reste fondateur :

- doctrine ;
- architecture ;
- stratégie ;
- accès sensibles ;
- supervision ;
- décisions structurelles.

---

## Interdits publics

Ne jamais exposer publiquement :

- PIN ;
- codes ;
- owner_id ;
- liens cockpit privés ;
- téléphone complet si non nécessaire ;
- données clients ;
- notes internes ;
- références paiement sensibles ;
- preuves Wave / Orange Money ;
- documents personnels ;
- informations famille ;
- données administratives ;
- historiques PAY.

---

## QR sécurisé

Le QR est une carte de circulation publique.

Il doit pointer vers :

- une fiche publique ;
- un service public ;
- une boutique publique ;
- un profil chauffeur public ;
- un logement public ;
- un lieu public ;
- une annonce publique.

Il ne doit jamais pointer vers :

- cockpit privé ;
- page PIN ;
- lien admin ;
- preuve de paiement ;
- donnée client ;
- activation sensible ;
- fichier interne.

---

## PAY protégé

PAY est une mémoire économique.  
PAY doit rester protégé.

Règles :

- aucune donnée PAY en public ;
- aucune référence sensible en QR public ;
- validation humaine avant entrée critique ;
- séparation recette / dépense / dette ;
- accès cockpit protégé ;
- admin seulement si nécessaire ;
- trace propre des actions.

---

## Oreille DIGIY et sécurité

L’Oreille DIGIY peut écouter, classer et préparer.

Elle ne doit pas :

- valider un paiement seule ;
- publier une fiche seule ;
- envoyer un message sensible sans validation ;
- supprimer une donnée seule ;
- exposer une donnée privée ;
- activer un accès sans contrôle.

**L’Oreille prépare.  
L’humain valide.**

---

## IA locale et sécurité

La future IA locale doit rester en soutien.

Elle peut :

- comprendre ;
- classer ;
- résumer ;
- reformuler ;
- préparer.

Elle ne doit pas décider seule.

**Le local prépare.  
Le cloud renforce.  
Le pro décide.**

---

## Anti-lookup

DIGIYLYFE doit être l’inverse des moteurs de recherche de données volées.

Eux font :

**fuite → compilation → exposition → monétisation**

DIGIY fait :

**accord → fiche choisie → utilité → protection → pro qui décide**

---

## Règle d’or

**Une donnée trouvée quelque part n’est pas une donnée libre à utiliser.**

Même si une information circule, DIGIY ne doit pas l’aspirer sans cadre.

---

## Résumé

DIGIYLYFE protège parce qu’il veut durer.

La confiance est un actif.  
La donnée est une responsabilité.  
Le pro garde la main.  
Le public garde sa dignité.  
Le système garde ses limites.

**Pas d’aspiration.  
Pas d’exposition inutile.  
Pas de décision automatique sensible.**
